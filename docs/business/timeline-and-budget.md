# RouteScout Timeline And Budget Proposal

## Purpose

This document proposes a near-term timeline and budget for RouteScout.

It is designed for founder alignment over the next 3 months.

## Planning Window

Proposed active build window:
- `12 weeks`

High-level objective:
- get the planner web app live quickly
- get the iPhone MVP running during the same window
- begin testing user acquisition and subscription conversion

## Delivery Strategy

The sequence should be:

### Stage 1: Planning Product First

Priority:
- get the planner web app up as soon as possible

Reason:
- fastest route to a usable product
- easiest place to validate the route-planning workflow
- best surface for refining plans, route cards, and brand

### Stage 2: iPhone MVP

Priority:
- ship the iPhone app soon after the planner is stable

Reason:
- the iPhone app is necessary for the full product story
- it is important for premium consumer perception
- it is likely the more valuable strategic asset long-term

## Proposed Timeline

### Weeks 1-2

Focus:
- finalize core product design
- finalize planner experience
- implement core web planner surface
- connect planning UI to route generation

Deliverables:
- working planner web app
- starter plans
- route generation and comparison
- branded route and spec cards

### Weeks 3-4

Focus:
- stabilize planner
- improve saved plans and saved routes
- improve route card system
- begin iPhone MVP implementation

Deliverables:
- cleaner web MVP
- clearer account / saved-item behavior
- iPhone app foundation

### Weeks 5-8

Focus:
- build iPhone MVP
- support route review and route use on mobile
- align visual system and product logic across surfaces

Deliverables:
- first meaningful iPhone app
- shared product system across web and iOS
- early end-to-end usable experience

### Weeks 9-12

Focus:
- QA and polish
- refine onboarding and subscription logic
- founder-led marketing
- early traction push

Deliverables:
- public-ready MVP
- initial subscriber test
- traction metrics for the next funding or strategic decision

## Budget Proposal

### Founder Build Compensation

Requested compensation:
- `$1,000/week`
- for `12 weeks`

Total requested build compensation:
- `$12,000`

Purpose:
- compensate active product, design, engineering, brand, and launch execution work
- reduce the imbalance created by a 50/50 equity split while labor is not actually equal

### Performance Bonuses

Requested milestone bonuses:
- `$5,000` at `100 activated subscribers`
- `$10,000` at `1,000 activated subscribers`

Purpose:
- tie upside to traction
- reward real market validation
- keep incentives aligned without requiring a larger up-front salary

### Total Near-Term Cash Exposure

Base 3-month compensation:
- `$12,000`

Potential milestone bonuses after traction:
- up to `$15,000`

Total possible if milestones are hit:
- `$27,000`

## Budget Framing

This proposal is modest relative to the actual workload if one founder is doing:
- product strategy
- UX
- brand
- frontend
- backend
- mobile app development
- marketing execution

The base weekly compensation should be framed as:
- partial compensation for concentrated build labor
- not a full market salary

## Why This Budget Is Reasonable

This budget is reasonable because:
- the product is not a hobby-level effort
- the work is cross-functional and founder-critical
- the execution burden is not evenly split
- the amount is small enough to be financeable
- the bonus structure shifts some upside to real traction events

## Budget Recommendation

Recommended founder proposal:

1. approve a `12-week` build window
2. approve `$1,000/week` during that period
3. approve milestone bonuses at:
   - `100 activated subscribers`
   - `1,000 activated subscribers`
4. reassess after the 12-week window based on:
   - shipped product
   - user traction
   - subscriber count
   - strategic interest

## Decision Points

At the end of the 12-week build window, founders should decide:

1. continue funding and expand
2. hold and iterate with a smaller burn
3. pursue partnerships or acquisition conversations
4. stop if traction is weak

## Recommended Tone For The Proposal

This should be presented as:
- practical
- founder-aligned
- modest but fair
- structured around execution and outcomes

It should not be framed as:
- a demand without milestones
- a vague future promise
- an assumption that equity alone covers all labor

## Bottom-Line Proposal

Proposed near-term deal:
- `12 weeks`
- `$1,000/week`
- web planner up first
- iPhone MVP during the same window
- `$5,000` bonus at `100 activated subscribers`
- `$10,000` bonus at `1,000 activated subscribers`
- reassess after the initial build phase
